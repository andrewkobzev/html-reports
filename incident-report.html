INC-247 — Cart submit failures on Mar 14

3-hour partial outage of the order entry submit path. ~190 affected orders queued and replayed.

Summary

Impact ~190 orders failed at submit step between 14:12 – 17:04 ET. All recovered via replay queue. No data loss.

Trigger MSSQL stored proc usp_lock_order hit a query plan regression after stats refresh.

Resolution Forced plan recompile via OPTION (RECOMPILE) hint, then queue drain.

Detected by Pager — Synthetic submit probe failed 5/5.

MTTD 4 min · MTTR 2 h 52 min

Why? (five whys)

  1. Submit endpoint timed out — because usp_lock_order took >30s instead of <100ms.
  2. The SP got slow — because the query plan switched to nested loop joins.
  3. The plan switched — because the stats refresh updated cardinality estimates outside the plan cache's tolerance.
  4. Cardinality drifted — because t_order_lock rows grew 8× last month with no maintenance follow-up.
  5. No one noticed the row growth — because we have no monitor on table size deltas after the queue subsystem changed.
