Feature explainer

How rate limiting works in this repo

TL;DR

Two-layer scheme. Per-IP token bucket at the edge (Nginx) for raw flood protection, then per-account sliding window inside the Node app for business-tier quotas. Configurable via env vars, not code.

The request path

01 Nginx receives the request

The first check is in the Nginx upstream block — token bucket keyed on $binary_remote_addr, sized for ~30 rps burst per IP.

limit_req_zone $binary_remote_addr zone=oe_edge:10m rate=30r/s;
location /orderentry/ {
  limit_req zone=oe_edge burst=20 nodelay;
  proxy_pass http://oe-backend;
}
02 Backend receives, validates JWT, attaches account

Auth middleware reads x-iisnode-auth_user (or Easy Auth header), looks up the account, attaches it to req.user. Rate limiter reads req.user.accountId next.

03 Per-account sliding-window check

In-memory Map keyed on accountId, sliding window of 60s. Default cap is 600 req/min/account. Hot accounts get override via RATE_OVERRIDES env JSON.

RATE_OVERRIDES='{"CU101-0000000001": 3000}'
04 Response — 429 with Retry-After

On hit, returns 429 Too Many Requests with Retry-After header set to seconds until window slides. Frontend swallows + retries silently up to 3 times.

Configuration

RATE_LIMIT_PER_MIN=600
RATE_OVERRIDES='{"CU101-...":3000}'
RATE_LIMIT_WINDOW_MS=60000
<add key="RATE_LIMIT_PER_MIN" value="600" />
<add key="RATE_OVERRIDES" value='{"CU101-...":3000}' />
az containerapp update \
  --name containerapp-oe-backend-sandbox \
  --set-env-vars RATE_LIMIT_PER_MIN=600

FAQ

What if I need to bypass the limit for an integration test?

Add the test client IP to RATE_BYPASS_IPS in .env. Bypass is logged at info level so audit still sees the calls.

Why in-memory and not Redis?

Single-instance Node app today. When we go multi-instance on Container Apps, the limiter is the first thing to swap for Redis. Tracked in BW-4220.

Does this apply to MCP tool calls?

No — MCP has its own API-key-based gate, lives outside this path. See mcp-server/README.md.

Glossary

Token bucket
Algorithm where each request consumes a token; tokens refill at a fixed rate. Allows short bursts.
Sliding window
Counts requests in the last N seconds, recomputed continuously. Smoother than fixed windows.
Burst
Maximum requests allowed above the steady rate before the limiter kicks in.
Retry-After
HTTP header the server returns with a 429, telling the client how many seconds to wait.
